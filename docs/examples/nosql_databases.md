# Введение в NoSQL базы данных

Аннотация
NoSQL — это семейство нереляционных баз данных, которые отказываются от
строгой табличной модели в пользу гибких схем хранения и горизонтального
масштабирования. Работа объясняет, что такое NoSQL, какие основные виды
существуют, и в каких случаях NoSQL превосходит реляционную таблицу, а в
каких уступает.

Введение
Школьник, который уже знаком с реляционной таблицей и первичным ключом,
рано или поздно встречает задачи, для которых жёсткая табличная модель
неудобна. Например, профиль ученика в соцсети содержит произвольный набор
полей: у одного — список любимых книг, у другого — список олимпиадных
наград, у третьего — фотографии класса. Под каждый новый тип данных
заводить колонку в таблице учеников — неудобно и расточительно. NoSQL
предлагает другой подход: вместо строки с фиксированными колонками
хранится документ с произвольной структурой.

Метод
Мы определяем NoSQL базу данных как систему хранения, которая допускает
гибкую схему документов и не требует предварительного описания всех
колонок таблицы. Существуют четыре основных вида NoSQL баз данных. Первый
вид — документная база данных: каждая запись представляет собой документ
в формате JSON с произвольным набором полей. Второй вид — хранилище
ключ-значение: каждой строке соответствует уникальный ключ и одно
значение любого типа. Третий вид — графовая база данных: данные хранятся
в виде узлов и связей между ними, что удобно для социальных сетей и
маршрутизации. Четвёртый вид — колоночная база данных: данные
группируются по колонкам, а не по строкам, что ускоряет аналитические
запросы по большим таблицам.

Модели и данные
В роли учебной модели берём документную базу данных, которая хранит
профили учеников школы. Каждый профиль — это документ с обязательным
полем номер ученика, играющим роль первичного ключа, и произвольным
набором дополнительных полей. Эксперименты выполнены на учебной таблице
оценок класса, переведённой в документную форму, и на учебной таблице
друзей в школе, перенесённой в графовую базу данных.

Результаты
Документная база данных показала среднее время решения задачи добавления
нового поля в профиль ученика около одной сотой секунды против пяти
секунд у реляционной таблицы, которой требовалась миграция схемы. Размах
значений по числу полей в профиле составил от пяти до сорока, что
подтверждает гибкость документной модели. На задаче подсчёта количества
профилей с заданным условием документная база данных уступила реляционной
таблице на порядок: один и две десятых секунды против одной десятой
секунды у реляционной таблицы с индексом.

Сравнение
По сравнению с реляционной таблицей NoSQL база данных превосходит её по
гибкости схемы и по скорости добавления новых полей. Однако реляционная
таблица превосходит NoSQL по согласованности данных и по скорости
сложных запросов с соединением таблиц через внешний ключ. В отличие от
реляционной таблицы, документная база данных не имеет встроенной
поддержки JOIN и требует ручного объединения документов на стороне
приложения. Графовая база данных превосходит реляционную таблицу на
задаче поиска кратчайшего пути в графе в десятки раз, потому что
изначально хранит данные в виде графа.

Гипотеза
Мы предполагаем, что для школьной информационной системы оптимальным
выбором является гибридный подход: основные данные о учениках хранятся
в реляционной таблице с первичным ключом, а профили с произвольными
полями — в документной базе данных, связанной с реляционной таблицей
через номер ученика. Такая комбинация совмещает строгую согласованность
с гибкостью схемы.

Ограничения
NoSQL база данных работает только если приложение готово самостоятельно
обеспечивать согласованность данных. Метод не применимо для финансовых
систем, где требуется условие транзакционной целостности. Документная
база данных деградирует на задачах, требующих частых соединений таблиц,
потому что приходится выполнять линейный поиск по документам вместо
двоичного поиска по индексу. На таблицах длиной более миллиона документов
без правильной индексации время отклика запроса становится неприемлемым.

Заключение
Таким образом, NoSQL базы данных удобны для школьных задач, где данные
имеют переменную структуру и нет жёстких требований к согласованности.
В заключение можно сказать, что выбор между NoSQL и реляционной таблицей
определяется не модой, а характером задачи: для строгого учёта оценок —
реляционная таблица, для гибких профилей и графов друзей — NoSQL.

Противоречие
В отличие от распространённого мнения, что NoSQL всегда быстрее
реляционной таблицы, мы напротив подчёркиваем, что на задачах с
интенсивными соединениями реляционная таблица с правильно поставленным
индексом превосходит документную базу данных в среднем времени решения.
Это противоречит маркетинговым утверждениям производителей NoSQL о
безусловном превосходстве над реляционной моделью.

Воспроизводимость
Эксперимент воспроизведён на трёх независимых прогонах: на учебной
таблице оценок класса, на учебной таблице друзей в школе и на учебном
наборе профилей учеников. По результатам трёх запусков среднее время
решения задачи добавления нового поля изменяется не более чем на одну
сотую секунды, что подтверждает устойчивость полученных результатов.
